# Feedback on AI Software Factory Design Draft

## Clarity
**Rating: High**
The design is articulate and easy to understand. The choice of using the file system as a state machine (Directory-Based State Machine) is intuitive and visual.
- The mapping between directories (`1created`, `2working`, etc.) and lifecycle stages is clear.
- The role of auxiliary directories (`100templates`, `300logs`) is well-defined.
- The pseudocode for the orchestration loop (`Execute-StoryPipeline`) effectively communicates the intended workflow.

## Feasibility
**Rating: High**
The proposed design is practically implementable with the current tech stack.
- **Core Tech**: PowerShell + File System is a robust, low-dependency choice for the orchestration layer.
- **AI Integration**: Using `gh copilot -p` is a standard CLI interaction.
- **Risk Mitigation**: The design correctly identifies "AI Output Instability" as a primary risk and proposes valid mitigations (JSON enforcement, schema validation).
- **Simplicity**: By avoiding complex databases or message queues in favor of atomic file moves, the system achieves high reliability with low complexity.

## Completeness
**Rating: Medium-High**
The design covers the majority of the lifecycle but leaves a few critical implementation details open:

1.  **Code Modification Strategy**:
    - The design mentions `Invoke-Implementation`, but does not specify *how* code changes are applied.
    - Does the AI output full files? Unified diffs? `sed` scripts?
    - **Recommendation**: Define a standard "Patch Format" (e.g., a JSON block containing `file_path`, `search_pattern`, `replace_content`) to ensure the orchestration script can reliably apply changes without syntax errors.

2.  **Context Management Strategy**:
    - `context.md` is mentioned, but the mechanism for populating it is not.
    - Simply dumping the repo into context will hit token limits.
    - **Recommendation**: Add a specific step or agent capability (e.g., `Invoke-ContextGathering`) that uses grep/glob or a vector index to populate `context.md` with *only* relevant files before planning starts.

3.  **Review Feedback Loop**:
    - The flow suggests `4review` -> `5approved` or implicit failure.
    - It is unclear how a "Request Changes" scenario is handled. Does it go back to `2working`?
    - **Recommendation**: Explicitly define the transition `4review` -> `2working` (with feedback appended to `context.md` or a new `feedback.md`) to allow for iterative fixes without restarting the story.
