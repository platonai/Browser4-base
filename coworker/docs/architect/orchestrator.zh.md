# AI 软件工厂编排器 (AI Software Factory Orchestrator)

## 概述

`orchestrator.ps1` 脚本是 AI 软件工厂的核心引擎。它通过一系列 AI 驱动的阶段（从分析到实施和验证）处理任务（用户故事），从而实现软件开发生命周期的自动化。

## 工作原理

### 工作流程

1.  **监控**：编排器持续监控 `coworker/tasks/1created` 目录中的新任务文件夹。
2.  **处理**：当检测到新任务时，它会被立即移动到 `coworker/tasks/2working`，以表示工作已开始。
3.  **管道执行**：任务经历一个顺序的 AI 驱动步骤管道：
    *   **分析**：使用模板 `analysis.prompt.md` 分析 `story.md` 需求。
    *   **规划**：使用 `plan.prompt.md` 创建结构化开发计划 (`plan.json`)。
    *   **子功能设计**：使用 `subfeature.prompt.md` 生成详细设计文档 (`design.md`)。此步骤旨在迭代计划中定义的子功能。
    *   **实施**：根据设计，使用 `implementation.prompt.md` 生成代码补丁 (`impl.patch`)。
    *   **E2E 测试**：使用 `e2e.prompt.md` 生成端到端 (E2E) 测试 (`e2e.kt`) 以验证实施。
    *   **验证**：使用 `validation.prompt.md` 验证生成的工件是否符合原始需求。
4.  **完成**：所有步骤成功执行后，任务文件夹将移动到 `coworker/tasks/3_1complete`。
5.  **故障处理**：如果任何步骤失败（例如，文件丢失、执行错误），任务将移动到 `coworker/tasks/3_5aborted` 进行人工检查。

### 目录结构

*   `coworker/tasks/100templates`：包含管道每个阶段的提示模板。
*   `coworker/tasks/300logs`：按故事 ID 组织，存储每个任务的详细执行日志。
*   `coworker/tasks/1created`：新任务放置的输入目录。
*   `coworker/tasks/2working`：活动处理目录。
*   `coworker/tasks/3_1complete`：成功完成任务的归档。
*   `coworker/tasks/3_5aborted`：失败或中止任务的归档。

## 设计原则

1.  **自动化状态机**：系统使用文件系统目录来表示任务的状态（已创建 -> 工作中 -> 完成/中止），无需复杂的数据库即可确保持久性和易于监控。
2.  **模板驱动智能**：所有 AI 交互均由 Markdown 模板管理。这允许在不修改核心编排器代码的情况下，轻松自定义和调整 AI 的行为。
3.  **模块化管道**：流程分解为离散、逻辑的步骤（分析 -> 计划 -> 设计 -> 代码 -> 测试），模仿标准软件工程工作流程。
4.  **可追溯性**：与 AI 的每次交互都会记录时间戳、完整提示和原始输出，为调试和改进提供完整的审计跟踪。
5.  **幂等性与恢复**：系统设计为通过隔离有问题的任务来处理故障，防止它们阻塞整个管道。

## 未来演进

1.  **真实 AI 集成**：将当前的占位符 `Invoke-Copilot` 函数替换为对 GitHub Copilot 或其他 LLM 提供商的实际 CLI 调用（例如 `gh copilot`）。
2.  **迭代优化循环**：实施反馈循环，验证失败触发自动重新规划或重新实施尝试。
3.  **并行处理**：支持并发执行多个子功能甚至多个故事，以增加吞吐量。
4.  **增强的错误处理**：添加复杂的重试机制、指数退避和更细粒度的错误报告。
5.  **动态配置**：允许通过任务文件夹中的配置文件覆盖每个任务的配置（例如，模型选择、特定 linter 规则）。
6.  **人工介入**：在关键阶段之前（例如，代码提交之前）引入人工审查和批准的暂停点。
